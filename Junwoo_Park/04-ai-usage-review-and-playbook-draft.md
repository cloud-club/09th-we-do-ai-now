# Week 5. 나의 AI 사용 점검 및 직무 맞춤 적용 전략 설계

## 1. 현재 나의 AI 사용 방식 점검

현재 내가 가장 자주 쓰는 AI는 GPT 웹버전에서 Thinking 모드를 켰을 때의 `extended thinking`이다. 단순히 검색을 대신하거나 답변을 빨리 받는 용도로 쓰기보다는, 실무 문제를 구조화하고 판단 속도를 높이는 작업 파트너처럼 쓰는 경우가 더 많다.

내 직무는 제조 DX 환경에서 비즈니스 요구를 실제로 운영 가능한 클라우드·인프라 구조로 바꾸는 Technical Architect에 가깝다. 그래서 AI를 붙이는 지점도 자연스럽게 정해진다. 장애나 설정 이슈를 볼 때 어디부터 확인해야 할지 정리하고, 변경 전에 빠진 검토 포인트가 없는지 점검하고, 복잡한 내용을 문서로 정리하는 데 AI를 가장 자주 사용한다. 결국 내 일에서 중요한 건 “빨리 답을 얻는 것”보다 “운영 가능한 판단으로 이어지는가”이기 때문이다.

실제 사용 장면도 꽤 분명하다. Redis 연결 문제, NFS 및 스토리지 구성, Harbor 인증 방식, ArgoCD 설정 이슈, Kubernetes 리소스 산정, 네트워크와 LB/Gateway 경로 해석처럼 운영 환경에 가까운 문제를 다룰 때 AI를 자주 켠다. 이때도 단순히 “원인이 뭐야?”라고 묻기보다, 코드·설정·인프라·네트워크 중 어디부터 봐야 하는지, 어떤 순서로 확인해야 하는지, 운영상 위험은 무엇인지까지 함께 정리하려는 편이다.

문서 작업에서도 AI를 많이 쓴다. GitHub 제출용 Markdown 문서, Confluence 스타일 설계 문서, Runbook, 공지문, 스터디 과제 초안, 기술 블로그 기획처럼 머릿속 생각을 구조화해 글로 바꿔야 할 때 특히 도움이 컸다. 먼저 AI로 논리 뼈대를 잡고, 그다음 내가 문장을 다듬는 방식이 지금은 꽤 익숙해졌다.

또 한 가지는 AI를 작업 시스템 안쪽으로 끌어들이려는 시도다. Obsidian + Git 기반 정리 체계, Codex와의 작업 흐름, STT 결과물 아카이빙 구조, `n8n` 같은 자동화 흐름을 고민하면서 AI를 그냥 쓰고 끝나는 도구가 아니라, 내 일하는 방식 안에 들어올 수 있는 구성 요소로 보고 있다.

정리하면 지금의 나는 AI를 학습 보조 도구로도 쓰지만, 그보다 더 강하게 실무 판단 보조 도구, 문서화 엔진, 구조 설계 파트너, 자동화 구상 도구로 활용하고 있다.

## 2. 지금까지의 사용에서 아쉬웠던 점

AI를 꽤 적극적으로 써왔지만, 아쉬움도 분명했다. 가장 크게 느낀 건 “답이 틀렸다”기보다 “그럴듯하지만 바로 실무에 붙이기에는 한 끗 모자라다”는 점이었다. 클라우드·인프라 쪽에서는 개념 설명만으로는 부족하고, 결국 명령어, 설정 예시, 검증 순서, 리스크 포인트처럼 바로 실행 가능한 형태가 필요하기 때문이다.

또 하나는 내 사용 습관에서 오는 아쉬움이었다. 나는 현재 문제를 해결하는 동시에 재발 방지 구조, 문서화 방식, 자동화 가능성, 아카이빙 체계까지 같이 보려는 편이다. 이 성향은 분명 장점이기도 하지만, 잘못하면 당장 해야 할 액션보다 구조 설계가 앞서면서 작업이 무거워진다.

돌아보면 특히 아래 네 가지가 자주 걸렸다.

- AI 답변이 설명으로는 충분한데, 실제 환경에 바로 적용할 수 있는 수준까지는 내려오지 않는 경우
- 배경 설명은 충분했지만 정작 내가 원하는 출력이 늦게 드러나서 답변이 넓게 퍼지는 경우
- 답변은 잘 받았는데, 로그·설정값·명령어 기준으로 검증하는 루프가 뒤늦게 붙는 경우
- 결과물을 잘 남기려는 의욕이 앞서서, 아직 덜 익은 흐름까지 너무 빨리 템플릿이나 구조로 만들려는 경우

결국 지금까지의 아쉬움은 AI를 덜 써서 생긴 문제가 아니라, 오히려 너무 넓고 깊게 활용하려다 작업 단위가 커진 데서 생긴 경우가 많았다고 생각한다.

## 3. 실패 패턴 분석

아래는 지금까지 반복적으로 드러난 실패 패턴을 정리한 것이다. 여기서 중요한 건 “무엇이 잘 안됐는가”보다 “왜 그 일이 반복됐는가”와 “다음에는 어떻게 바꿀 것인가”다.

| 실패 패턴 | 실제로 생긴 일 | 다음부터의 원칙 |
| --- | --- | --- |
| 문제 해결과 체계 설계를 동시에 진행하는 패턴 | 당장 해결해야 하는 운영 이슈를 보면서 동시에 문서 템플릿, 재발 방지 구조, 자동화 흐름까지 같이 설계하려다 우선순위가 흐려졌다. | 먼저 지금 해결할 문제와 나중에 구조화할 항목을 분리한다. 세션 안에서는 실행 단위를 먼저 끝낸다. |

| 답변은 잘 받았지만 검증 루프가 늦게 붙는 패턴 | 구조화된 좋은 답을 받은 뒤에야 로그, 설정값, 명령어 기준으로 검증 단계를 붙이는 일이 있었다. | 처음부터 `가설 → 확인 명령 → 기대 결과 → 다음 분기` 구조로 답을 받는다. |
| 재사용 구조를 너무 이르게 만들려는 패턴 | 아직 충분히 반복되지 않은 흐름도 곧바로 템플릿, 스킬 파일, 폴더 구조, 자동화 대상으로 만들고 싶어졌다. | 최소 몇 번은 실제로 반복된 뒤에야 템플릿화하거나 자동화 후보로 올린다. |

## 4. 내 직무에 맞는 AI 적용 전략 설계

클라우드·인프라 엔지니어인 내 업무에서 AI가 가장 먼저 들어가야 할 곳은 운영 환경을 직접 바꾸는 영역이 아니다. 오히려 판단, 분석, 문서화, 점검 순서 정리처럼 사람의 판단 품질과 속도를 높이는 곳이 먼저라고 본다. 이유는 간단하다. 이 일은 한 번의 잘못된 변경이 바로 장애로 이어질 수 있기 때문이다.

### 1) 1순위: 장애/이슈 분석 보조

가장 먼저 붙여야 할 영역은 장애 분석이다. 실제로 Redis 연결 실패, Storage 이슈, ArgoCD 동기화 문제, Harbor 인증 문제, 네트워크 경로 해석 같은 작업에서 AI의 체감 효과가 가장 컸다. 이때 AI의 역할은 장애를 대신 해결하는 것이 아니라, 원인 후보를 분류하고 점검 순서를 제안하는 것이다.

내가 기대하는 출력은 대체로 이렇다.

- 원인 후보를 코드, 설정, 인프라, 네트워크 관점으로 나눠주기
- 어디부터 배제해야 할지 점검 순서 제안하기
- 가설별 확인 방법과 기대 결과 정리하기

### 2) 2순위: 운영 문서 및 Runbook 작성

실무에서는 실제 작업보다 기록이 더 오래 남는다. 그래서 운영 문서, 점검 체크리스트, 인수인계 문서, Runbook 초안 작성은 AI를 붙이기 좋은 영역이다. 위험도는 낮고, 재사용성은 높고, 축적 가치도 크다. 특히 회의 메모나 작업 로그가 남아 있을 때 이를 운영 문서 형태로 정리하는 데 AI가 꽤 유용했다.

### 3) 3순위: 변경 전 영향도 검토

클라우드·인프라 업무는 작업 중 사고보다 작업 전에 충분히 생각하지 못해서 나는 사고가 더 많다. Storage 구조 변경, Gateway 정책 변경, ResourceQuota 조정, Redis 구성 변경 같은 작업에서는 AI를 “사전 리뷰어”로 두는 게 가장 현실적이다.

여기서 필요한 건 정답이 아니라 다음과 같은 정리다.

- 빠지기 쉬운 사전 확인 항목
- 영향 받을 수 있는 시스템
- 롤백 시 같이 봐야 할 포인트
- 승인 전에 확인해야 할 위험 요소

### 4) 4순위: 반복 운영 절차의 템플릿화와 자동화 설계

자동화는 중요하다. 다만 처음부터 운영계 직접 조치 자동화로 가는 건 맞지 않다고 생각한다. 먼저 반복 작업을 식별하고, 입력과 출력이 무엇인지 정리하고, 사람이 승인하기 전 단계까지 자동화하는 흐름이 현실적이다. 지금 내가 관심을 두는 `CLI`, `Codex`, `Git`, `Obsidian`, `n8n`도 결국 이 흐름 안에서 쓰이는 게 맞다.

### 5) 원칙: 운영계 직접 변경은 가장 나중

AI가 운영 클러스터에 직접 변경을 적용하거나, 승인 없이 설정을 바꾸거나, 자동으로 배포와 롤백을 수행하는 영역은 지금 단계에서 가장 나중에 가야 한다. 내 기준에서 AI의 초반 역할은 “자동 실행”이 아니라 “판단 품질과 속도 향상”이다.

지금 내 직무에 맞는 성숙 순서를 한 줄로 정리하면 이렇다.

> 장애 분석 → 변경 검토 → 문서화 → 자동화 설계 → 제한적 자동 실행

## 5. 앞으로 해보고 싶은 AI 활용 디벨롭 방향

앞으로는 AI를 더 많이 쓰는 것보다, 지금 실제로 자주 하는 작업에 더 깊게 붙이는 쪽이 중요하다고 생각한다. 지금 기준에서 가장 집중하고 싶은 방향은 세 가지다.

### 1) CLI 중심 활용 고도화

웹형 AI는 사고를 정리할 때는 편하지만, 실제 파일 수정, Git 작업, 로컬 문서 편집까지 이어지는 실행력은 CLI 기반 환경이 더 좋다. 그래서 앞으로는 로컬 문서 초안 작성, Git 흐름, 파일 단위 편집, 작업 로그 정리를 CLI 쪽으로 더 자연스럽게 묶고 싶다.

### 2) 문서/점검용 프롬프트 템플릿화

지금 내가 자주 하는 일은 운영 문서 작성, 변경 전 점검, 장애 분석, 회의 내용 정리처럼 구조가 비슷한 작업들이다. 이 부분은 매번 처음부터 설명하기보다 입력 형식을 고정한 템플릿으로 바꾸는 게 훨씬 효율적이다. 장애 분석용, 변경 검토용, Runbook 작성용, 점검 체크리스트용 프롬프트를 따로 정리해두고 싶다.

### 3) Runbook/점검 절차 초안 자동화

반복적으로 만드는 운영 절차 문서와 점검 흐름은 AI가 초안화하기에 가장 적합한 영역이다. 회의 메모, 로그, 작업 이력, 명령어 결과를 바탕으로 Runbook 초안과 점검 절차 초안을 빠르게 만드는 흐름을 붙이면, 문서화 시간을 줄이면서 기준도 더 일관되게 가져갈 수 있을 것 같다.

지금은 이 세 가지를 따로 보는 게 아니라 하나의 흐름으로 보고 있다. 먼저 CLI 기반 실행 환경을 조금 더 다듬고, 그 안에서 문서와 점검용 프롬프트를 템플릿화하고, 마지막으로 Runbook과 점검 절차 초안 생성을 자동화하는 방향이 가장 현실적이다.

## 6. 나의 AI Playbook 초안

### 6-1. AI를 먼저 붙일 업무

- 장애/이슈 분석
- 변경 전 영향도 검토
- 운영 문서 및 Runbook 작성
- 구조 설계 초안 정리
- 반복 작업의 자동화 후보 식별

운영계 직접 변경이나 자동 실행은 사람 검토와 승인 전까지 AI에 맡기지 않는다.

### 6-2. 입력 방식

AI에게 질문할 때는 가급적 아래 네 줄 구조를 먼저 맞춘다.

- 현재 상황
- 이미 확인한 것
- 제약 조건
- 원하는 출력 형식

예를 들어 장애 분석이면 `현상 / 영향 범위 / 관련 로그 / 이미 확인한 내용 / 원하는 정리 방식` 정도는 먼저 정리해서 넣는다.

### 6-3. 출력 방식

업무 유형에 따라 받고 싶은 출력 형식도 어느 정도 고정해두려고 한다.

- 장애 분석: `가설 / 확인 방법 / 기대 결과 / 다음 분기`
- 변경 검토: `위험 요소 / 사전 확인 / 영향 범위 / 롤백 포인트`
- 문서화: `배경 / 목적 / 현재 상태 / 조치 / 검증 / 운영 고려사항`
- 자동화 검토: `반복 작업 / 입력값 / 출력값 / 승인 필요 여부 / 자동화 가능성`

### 6-4. 검증 원칙

AI의 답변은 그대로 실행하지 않는다. 최소한 아래 순서로 본다.

1. 사실 검증: 설정값, 버전, 옵션, 리소스명이 실제 환경과 맞는지 확인
2. 절차 검증: 순서가 맞는지, 빠진 선행 조건이 없는지 확인
3. 위험 검증: 운영 반영 시 영향 범위와 장애 가능성 확인
4. 근거 검증: 최신성이 중요한 주제는 공식 문서와 레퍼런스로 다시 확인

### 6-5. 도구 사용 기준

- GPT 웹버전의 Thinking 모드 `extended thinking`: 학습, 문제 구조화, 문서 초안, 전략 정리에 가장 먼저 사용
- `Codex`나 CLI 기반 AI: 로컬 문서 편집, Git 작업, 실제 파일과 연결된 초안 생성에 사용
- 비교형 사용: 관점 차이나 표현 차이가 궁금할 때 보조적으로 사용
- Workflow 형태: 반복 문서화, 회고, 점검 요청처럼 입력과 출력이 정리된 뒤 적용
- Agent 형태: 아직은 탐색적 활용에 한정하고, 운영계 직접 변경 권한은 주지 않음

### 6-6. 세션 종료 원칙

하나의 작업 세션이 끝나면 최소한 아래 다섯 가지는 남긴다.

- 세션 목적
- 결정된 내용
- 미해결 이슈
- 다음 액션
- 참고 로그, 파일, 링크

이걸 남겨두면 다음 세션에서 처음부터 다시 설명하는 일이 훨씬 줄어든다.

### 6-7. 아카이빙 원칙

결과물은 가능하면 세 층으로 나눈다.

- 실행본: 실제로 바로 쓰는 최종 결과
- 판단본: 왜 그렇게 결정했는지 남기는 근거
- 원본: 대화, 로그, 참고자료, 초안

이렇게 나눠두면 나중에 다시 찾을 때 “무엇을 했는지”뿐 아니라 “왜 그렇게 했는지”까지 같이 볼 수 있다.

### 6-8. 자주 쓰는 템플릿

#### 장애 분석 템플릿

```text
문제 현상:
발생 시점:
영향 범위:
이미 확인한 사항:
관련 로그/설정:
원하는 출력:
- 가장 유력한 가설 3개
- 각 가설별 확인 명령어
- 기대 결과
- 다음 분기
- 적용 전 위험 포인트
```

#### 변경 영향도 분석 템플릿

```text
변경 대상:
변경 목적:
현재 구조:
예상 영향 범위:
제약 조건:
검토받고 싶은 항목:
- 주요 위험 요소
- 사전 확인 항목
- 영향 받을 시스템
- 권장 작업 순서
- 롤백 시 고려사항
```

#### Runbook 초안 생성 템플릿

```text
작업/이슈명:
배경:
현재 상태:
수행한 조치:
검증 결과:
운영 시 주의사항:
원하는 출력:
- 목적
- 절차
- 확인 포인트
- 실패 시 대응
- 인수인계용 메모
```

#### 세션 종료 요약 템플릿

```text
세션 목적:
이번에 결정된 내용:
미해결 이슈:
다음 액션:
참고 파일/링크:
```

### 6-9. 자동화 아이디어

- 장애 분석 입력 템플릿과 로그 요약 흐름을 표준화해서 초동 대응 정보를 자동으로 정리하기
- 회의 메모, 작업 로그, 명령어 결과를 Runbook 초안으로 변환하는 문서화 보조 흐름 만들기
- 변경 작업 전에 영향도 분석 체크리스트를 자동 생성해서 누락 줄이기
- 세션 종료 요약을 Obsidian과 Git 구조에 맞게 저장하는 아카이빙 흐름 만들기
- 반복 점검 절차를 수집해 자동화 후보 목록으로 관리하고, 충분히 검증된 절차만 스크립트화하기

장기적으로는 AI를 단순한 질의응답 도구가 아니라, 내 업무의 입력-분석-검토-문서화-기록-자동화 준비 단계에 연결된 시스템 구성 요소로 키우고 싶다.

- 운영 문서, 점검 체크리스트, 장애 분석, 변경 검토에 대해 반복 사용 가능한 프롬프트 템플릿 세트 정리하기
- Runbook 및 점검 절차 초안 생성을 작은 자동화 흐름으로 연결해, 메모와 로그를 재사용 가능한 문서 초안으로 바꾸는 방식 실험해보기
