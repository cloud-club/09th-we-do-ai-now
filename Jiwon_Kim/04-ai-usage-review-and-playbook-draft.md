### 1-2. 자주 사용하는 패턴

**가장 자주 하는 AI 활용 방식:**
장문의 맥락 (JD, 과제 요구사항, 기존 문서)을 통째로 붙여넣고, 구조화된 결과물(문서, 슬라이드 구성, 답변 초안)을 한 번에 요청하는 방식.

**이 방식을 자주 쓰는 이유:**
편하게 빠른 답변을 받기 위해서 

**현재 방식의 장점:**
- 빠르게 초안 확보 가능
- 구조 설계 시간을 크게 단축
- 기술 용어/포맷 일관성 유지에 유리

**현재 방식의 한계:**
- 맥락을 매번 새로 붙여넣어야 해서 반복 비용이 큼
- AI 출력을 그대로 쓰면 "내 목소리"가 사라지는 경향
- 긴 대화일수록 앞 맥락이 희석되어 품질이 떨어짐

## 2. 지금까지의 사용에서 아쉬웠던 점

### 2-1. 비효율 또는 한계

**가장 아쉬웠던 점:**
매 과제마다 컨텍스트를 처음부터 재구성해야 한다는 점. 며칠에 걸친 작업에서 이전 대화 내용이 단절되면 일관성 유지가 어려웠음.

**결과 품질 측면의 문제:**
- 프롬프트가 모호하면 AI가 "그럴듯하지만 내 상황과 다른" 결과를 생성

**컨텍스트 관리 측면의 문제:**
- 과제 관련 정보(요구사항, 이전 결정사항, 제약 조건)가 대화 밖에 흩어져 있어 일관된 활용이 어려움
- 대화가 길어질수록 초반에 설정한 가정들이 희석됨

**업무 연결 측면의 문제:**
- AI 출력 → PPT/Word 변환까지 수동 작업이 여전히 많음
- 리서치 결과를 정리해두지 않아 유사한 검색을 반복하게 됨

---

### 2-2. 단발성 사용으로 끝난 이유

**왜 반복 가능한 방식으로 정착되지 않았는가:**
- 과제마다 요구사항이 달라 "이번에만 쓰는" 방식으로 접근하게 됨
- 좋은 프롬프트를 만들어도 저장하지 않아 다음에 재활용이 안 됨

**어떤 부분이 습관화/체계화되지 않았는가:**
- 프롬프트 라이브러리 없음 (잘 된 프롬프트를 그냥 흘려보냄)
- 결과 검증 체크리스트 없음 (직관으로만 판단)
- 도구별 역할 분담 기준 없음 (그때그때 느낌으로 도구 선택)

## 3. 실패 패턴 분석

### 3-1. 기대만큼 잘 되지 않았던 경험

**사례:**
과제에서 AWS 아키텍처 다이어그램 구성을 요청했을 때 

**당시 기대했던 결과:**
사용자 상황을 고려한 AWS 아키텍처 다이어그램 구성 

**실제 결과:**
일반적인 AWS 모범 사례 수준의 답변. 구체 설정값이 빠진 표면적 구성.

**왜 그렇게 되었는가:**
프롬프트에 제약 조건을 충분히 포함하지 않았고, AI를 "검색 엔진처럼" 사용한 결과.

---

### 3-2. 실패 원인 정리

| 실패 요인 | 구체 내용 |
|---|---|
| **프롬프트 문제** | 제약 조건 미명시, 역할 부여 없이 막연한 요청 |
| **컨텍스트 문제** | 과제 배경(회사 규모, 기술 스택, 예산)을 매번 충분히 제공하지 않음 |
| **검증 문제** | AI 출력 → 즉시 사용, 기술적 타당성 별도 확인 생략 |
| **도구 선택 문제** | 모든 작업을 Claude 단일 도구로 처리하려는 경향 |
| **사용 습관 문제** | 좋은 결과가 나와도 프롬프트/흐름을 기록하지 않음 |

---

### 3-3. 앞으로의 개선 원칙

1. **제약부터 명시한다**: 역할(Role), 목적(Goal), 제약(Constraint), 출력 형식(Format)을 항상 프롬프트 앞단에 명시한다.
2. **AI 출력은 초안이다**: 기술 판단이 필요한 내용은 반드시 공식 문서나 실사례로 교차 검증한다.
3. **잘 된 프롬프트는 반드시 저장한다**: Notion 또는 마크다운 파일로 프롬프트 라이브러리를 구축한다.

## 4. 내 직무에 맞는 AI 적용 전략

### 4-1. 내 직무와 핵심 업무

**나의 직무:** Solutions Architect / Customer Success (엔터프라이즈 AI·클라우드)

**핵심 업무:**
- 고객 요구사항 파악 → 기술 아키텍처 설계 및 제안
- 솔루션 PoC/데모 구성 및 기술 검증
- 온보딩, 도입 후 성과 측정 및 리포팅
- 내부/외부 기술 문서 및 제안서 작성

**AI가 가장 먼저 들어가야 할 업무:**
고객 제안서 초안 작성 및 기술 아키텍처 문서화 → 시간 대비 효과가 가장 크고, 반복 구조가 명확한 영역.

---

### 4-2. AI 적용 우선순위

1. **제안서/기술 문서 작성 자동화** — 고객사별 컨텍스트를 템플릿화하여 초안 생성 시간 단축
2. **기술 리서치 및 비교 분석** — 경쟁 솔루션 비교, 신기술 트렌드를 구조화된 형태로 정리
3. **고객 커뮤니케이션 초안 생성** — 미팅 후 팔로업 메일, 기술 Q&A 답변 초안 작성

---

### 4-3. 나에게 맞는 활용 방식

| 방식 | 나의 활용 |
|---|---|
| **웹형 대화** | 제안서 구조 설계, 인터뷰 준비, 아이디어 발산 |
| **비교형 사용** | GPT vs Claude로 동일 아키텍처 설계 요청 후 결과 비교 → 빠진 관점 보완 |
| **문서/지식 아카이빙 연계** | Notion에 프롬프트 라이브러리 + 고객사별 컨텍스트 문서 유지 |
| **Agent/Workflow 확장** | 고객 온보딩 에이전트, 기술 Q&A 자동 응답 등 — 직접 구축 경험을 포트폴리오화 |

---

### 4-4. 왜 이 방식이 내 직무에 맞는가

**직무 특성과의 연결:**
SA/CS는 기술 깊이와 커뮤니케이션 속도가 동시에 요구되는 직무. AI를 쓰면 깊이 있는 초안을 빠르게 확보할 수 있어 두 요구를 동시에 충족 가능. 특히 고객 앞에서 실시간으로 아키텍처 옵션을 비교 설명해야 할 때, 사전에 AI로 비교 분석을 준비해두면 설득력이 크게 높아짐.

**기대 효과:**
- 제안서 초안 작성 시간 50% 단축
- 기술 리서치 → 정리 → 공유 사이클 단축
- 에이전트 구축 경험을 실제 고객 가치로 전환 가능

**필요한 전제 조건:**
- 고객사별 컨텍스트 문서(요구사항, 기술 스택, 제약 조건) 사전 정리
- 프롬프트 라이브러리 구축 및 유지
- AI 출력 검증 루틴 확립

## 6. 나의 AI Playbook 초안

### 6-1. 나의 AI 활용 원칙

1. **제약 조건 먼저**: 역할(Role), 목적(Goal), 제약(Constraint), 출력 형식(Format)을 항상 프롬프트 앞단에 명시한다.
2. **초안 취급**: AI 출력은 60점짜리 초안이다. 기술 판단이 필요한 내용은 공식 문서·실사례로 반드시 교차 검증한다.
3. **자산화**: 잘 된 프롬프트, 유용한 출력 구조는 그 자리에서 Notion에 저장한다. 흘려보내지 않는다.

### 6-2. 도구별 역할 분담

| 도구 | 역할 |
|---|---|
| **GPT (4o)** | 빠른 아이디어 발산, 간단한 검색성 질문, 코드 스니펫 |
| **Claude** | 장문 문서 작성, 논리 구조 설계, 긴 컨텍스트 기반 분석, 코드 실행이 필요한 과제 |

| **CLI/API** | 반복 작업 자동화, 배치 문서 생성, 평가 파이프라인 |

### 6-3. 검증 루틴

| 검증 유형 | 방법 |

| **실행 가능성 검증** | 제안 내용이 고객의 예산·일정·기술 역량 안에서 실현 가능한지 체크 |

### 6-4. 반복 사용할 템플릿

**템플릿 1 — 기술 아키텍처 설계 요청**
```
역할: 너는 엔터프라이즈 AWS 솔루션 아키텍트야.
목적: [고객사 특성]에 맞는 [솔루션명] 아키텍처를 설계해줘.
제약: 예산 [X], 운영 인력 [Y명], 현재 기술 스택 [Z].
RTO/RPO 요구사항: [구체 수치].
출력 형식: 1) 아키텍처 개요 2) 서비스 선택 근거 3) 고려한 대안 및 탈락 이유
```

**템플릿 2 — 고객 제안서 초안 작성**
```
역할: 너는 B2B 기술 제안서 전문 작가야.
목적: [고객사명]을 위한 [솔루션명] 도입 제안서 초안 작성.
고객 배경: [산업, 규모, 현재 문제].
우리의 강점: [차별화 포인트].
출력 형식: 1) 고객 문제 정의 2) 솔루션 개요 3) 기대 효과 4) 도입 로드맵
```

**템플릿 3 — 기술 트렌드 리서치 정리**
```
역할: 너는 엔터프라이즈 AI 시장 분석가야.
목적: [기술/트렌드명]의 현황과 SA/CS 직무 관점에서의 시사점 정리.
출력 형식: 1) 핵심 정의(3줄) 2) 주요 플레이어 3) 고객에게 줄 수 있는 가치 4) 알아야 할 기술 용어
```
